# Homework_7 Docker та контейнери

## Варіант A. Аналіз життєвого циклу контейнера

## Завдання 1. Запуск контейнера

```
Для демонстрації було використано контейнер з Python HTTP-сервером.
```
### Команда запуску:
```bash
docker run -d --name webtest -p 8080:8000 python:3.12-slim python -m http.server 8000
```
### Перевірка запущеного контейнера:
```bash
docker ps
```

### Коментар:

```
Було створено та запущено контейнер webtest. Усередині контейнера працює простий HTTP-сервер Python, а порт 8000 контейнера проброшений на порт 8080 хостової системи.
```

## Завдання 2. Аналіз процесів контейнера

### Перегляд процесів усередині контейнера:
```bash

docker exec webtest sh -c "cat /proc/1/cmdline"

```
### Результат:
```
PID 1 усередині контейнера:

python -m http.server 8000
```

### Пояснення:
```
У Docker контейнер існує доти, доки працює процес з PID 1. У даному випадку це HTTP-сервер Python, який був запущений як основна команда контейнера.
```

## Завдання 4. Робота з логами

### Перегляд логів контейнера:
```bash
docker logs webtest
```
### Для генерації записів у логах було виконано:
```bash
curl http://localhost:8080
```
### Після чого повторно переглянуто логи:
```bash
docker logs webtest
```
### Пояснення:
```
Команда docker logs показує дані зі стандартного виводу (stdout) та стандартного потоку помилок (stderr) процесу контейнера.
```

## Завдання 5. Завершення роботи контейнера

### Зупинка контейнера:
```bash
docker stop webtest
```
### Перевірка стану:
```bash
docker ps -a
```
### Видалення контейнера:
```bash
docker rm webtest
```
### Пояснення:
```
Команда docker stop надсилає процесу PID 1 сигнал SIGTERM.

Якщо процес завершується коректно, контейнер зупиняється.

Якщо процес ігнорує сигнал SIGTERM, Docker через певний час надсилає сигнал SIGKILL, який примусово завершує процес.
```

### Висновок:
```
Під час виконання роботи було розглянуто життєвий цикл Docker-контейнера. Було запущено HTTP-сервер у контейнері, визначено процес PID 1, перевірено роботу застосунку та переглянуто логи контейнера. Також було досліджено механізм завершення контейнера через сигнали SIGTERM та SIGKILL.
```
### Головний висновок: 
```
Контейнер існує лише доти, доки працює його основний процес (PID 1). Після завершення цього процесу контейнер автоматично завершує свою роботу.
```
